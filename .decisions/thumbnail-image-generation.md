# Decision

## Title

Auto Thumbnail Generation with OpenAI gpt-image-2

## Date

2026-06-24

## Status

Accepted

## Decision

Articles can set `[thumbnail] mode = "auto"`. The metadata generator then
produces one thumbnail image per post id, shared by both locales, and rewrites
the field to `mode = "file"`, `path = "/thumbnails/<post-id>.png"`,
`generated = true`.

- Images are generated by OpenAI `gpt-image-2` at `size = "1024x1024"`,
  `quality = "medium"`, configured under `[thumbnail_generation]` in
  `src/content/metadata.toml`. Model, size, quality, and prompt files are all
  config-driven so they change without code edits.
- A new `openai` provider is used for images only. Tags and summaries stay on the
  free-tier Gemini provider. `metadata.toml` already carries per-section
  `provider`/`model` fields, so this extends the existing pluggable design.
- The image prompt lives at `src/content/prompts/thumbnail-generation.md` with a
  `{CONCEPT}` placeholder. The concept is an explicit `[thumbnail].concept`
  string when present, otherwise it is derived from the resolved Japanese
  `[sumup]` summary by Gemini using `src/content/prompts/thumbnail-concept.md`.
- Images are stored at `public/thumbnails/<post-id>.png` and referenced as
  `/thumbnails/<post-id>.png`. Generation is idempotent: an existing file is
  reused instead of regenerated.
- `.github/workflows/generate-metadata.yml` runs generation on pushes to `main`
  and commits resolved metadata and thumbnails with `[skip metadata][skip
  translate]`.

Image generation requires the billing-enabled `OPENAI_API_KEY`. The static
`--check` verification never calls any provider.

## Context

Tag and summary automation already existed (Gemini, free tier), but thumbnails
only resolved to a shared placeholder. Issue work called for real per-article
thumbnails and an automation workflow so pushing an article resolves all of its
metadata without a manual local run.

The user chose `gpt-image-2` at medium quality after comparing Gemini image
models and OpenAI options on cost and instruction-following. The thumbnail style
is a dense negative-constraint risograph prompt, which favors a model with strong
instruction adherence.

## Alternatives

- Google Imagen / Gemini image models: keeps a single provider and key, but the
  user preferred OpenAI's instruction-following for the constrained prompt.
- Generate the concept from the full article body: rejected because the summary
  is already produced by Gemini, so reusing it avoids a second long-context read.
- Store images outside `public/` or with non-deterministic names: rejected
  because the quality checks require a public path, a leading `/`, an existing
  file, and matching ja/en paths. A post-id-derived name satisfies all of them.
- A runtime/build-time image service: rejected; the site is static and images
  should be committed and reviewable.

## Reason

Deriving the image name from the post id makes the ja and en thumbnails match by
construction and makes generation idempotent. Sourcing the concept from the
Japanese summary keeps the image deterministic regardless of locale processing
order and reuses an already-generated short text instead of re-reading the body.
Config-driven model/size/quality keeps the model choice a one-line change if
pricing or availability shifts.

## Consequences

- Thumbnail generation is not free: it needs a billing-enabled `OPENAI_API_KEY`
  locally (`.op.env`) and as an Actions secret for the workflow. Every new post
  defaults to `mode = "auto"`, so each new article incurs one image generation.
- All generated images carry OpenAI's SynthID watermark.
- The generation workflow must restrict its commit to article frontmatter and
  `public/thumbnails/**`, and guard against looping with `[skip metadata]`.
- The concept step adds one free Gemini call per new post when no explicit
  concept is set.

## Revisit Conditions

Revisit if image cost becomes significant, if a different image model or provider
is preferred, if thumbnails need per-locale variants, or if the exact
`gpt-image-2` model id / `quality` enum changes upstream.
