# Plan

## Goal

Automate article metadata generation in CI and add AI thumbnail image
generation, so that pushing an article resolves its date, tags, summary, and a
generated thumbnail without manual local runs.

## Scope

1. Add an automation workflow that runs metadata generation on `main` pushes.
2. Add thumbnail image generation via OpenAI `gpt-image-2`:
   - model selection (gpt-image-2, square, medium quality, config-driven)
   - prompt storage location
   - generated image storage location
   - a `thumbnail.mode = "auto"` mechanism that wires the generated image into
     `post.thumbnail`

## Non-goals

- Changing the existing tag/summary generation behavior (Gemini, free tier).
- Migrating to Imagen or another provider.
- Changing how thumbnails are rendered in the UI (the loader already exposes
  `post.thumbnail`).
- Batch API usage.

## Assumptions

- Image generation requires a billing-enabled key; the existing free-tier
  `GEMINI_API_KEY` stays for text (tags/summary), and a new `OPENAI_API_KEY`
  is issued for images.
- One thumbnail image per post directory, shared by `index.ja.md` and
  `index.en.md` (the quality check already requires the two locales to match).
- gpt-image-2 REST: `POST https://api.openai.com/v1/images/generations`,
  `Authorization: Bearer $OPENAI_API_KEY`, body `{ model, prompt, size, quality }`,
  response image at `data[0].b64_json` (base64 PNG). Exact `quality` enum value
  ("medium") confirmed against the API at implementation time.

## Decisions

- **Model**: `gpt-image-2`, `size = "1024x1024"` (square), `quality = "medium"`.
  All three are config-driven in `metadata.toml` so they can change without code
  edits.
- **Provider**: new `openai` provider for images only. `metadata.toml` already
  carries per-section `provider`/`model` fields, so this extends the existing
  pluggable design rather than working around it.
- **Prompt storage**: `src/content/prompts/thumbnail-generation.md` (the
  risograph style prompt with a `{CONCEPT}` placeholder), referenced from
  `metadata.toml` `[thumbnail_generation].prompt_file` — same pattern as
  `tag-generation.md` / `summary-generation.md`.
- **Concept resolution** (fills `{CONCEPT}`):
  1. `[thumbnail].concept = "..."` explicit field if present, else
  2. a short concept phrase generated by the free Gemini text model from the
     resolved `ja` **summary (`sumup`)** text, using
     `src/content/prompts/thumbnail-concept.md`.
  The `ja` sibling's summary is always the concept source: it is short, so Gemini
  reads the long article body only once (during summary generation) and the
  concept call operates on that short text. This also makes the result
  deterministic regardless of locale processing order. Dependency: the summary
  must be resolved before the concept, so thumbnail generation runs after
  summary resolution in the pipeline.
- **Generated image storage**: `public/thumbnails/<post-id>.png`, referenced as
  `/thumbnails/<post-id>.png`. Satisfies the existing quality constraints
  (public path, starts with `/`, file exists, ja/en paths match because the name
  derives from the post id).
- **Mechanism**: new `thumbnail.mode = "auto"`. The resolver generates the image
  once per post id (idempotent: skip if the file already exists), then rewrites
  the field to `mode = "file", path = "/thumbnails/<id>.png", generated = true`.
  `mode = "none"` keeps its current default-placeholder behavior (backward
  compatible).

## Steps

### Config & prompts
1. `src/content/metadata.toml`: add
   ```toml
   [thumbnail_generation]
   provider = "openai"
   model = "gpt-image-2"
   size = "1024x1024"
   quality = "medium"
   prompt_file = "src/content/prompts/thumbnail-generation.md"
   concept_prompt_file = "src/content/prompts/thumbnail-concept.md"
   ```
   Keep the existing `[thumbnail].default_path`.
2. Add `src/content/prompts/thumbnail-generation.md` (the risograph prompt with
   `{CONCEPT}`) and `src/content/prompts/thumbnail-concept.md` (concept phrase
   instruction).
3. `public/thumbnails/.gitkeep` so the directory exists.

### Schema & config types
4. `scripts/metadataSchema.mjs`: allow `thumbnail.mode = "auto"`; allow optional
   `thumbnail.concept` string.
5. `scripts/metadataConfig.mjs`: extend the typedef for `thumbnail_generation`.

### Generator
6. `scripts/generate_metadata.mjs`:
   - Add `openaiGenerateImage({ apiKey, model, prompt, size, quality })` using the
     same `fetch` style as `geminiGenerateJson`; return PNG bytes from
     `data[0].b64_json`.
   - Add concept resolution: explicit `[thumbnail].concept` field, else Gemini via
     `thumbnail-concept.md` fed the resolved `ja` `sumup` text (not the article
     body). Run thumbnail resolution after summary resolution so the summary is
     available.
   - Replace `resolveDefaultThumbnail` handling so `mode = "auto"` generates the
     image (idempotent on the output path), writes `public/thumbnails/<id>.png`,
     and rewrites the metadata to a resolved `file` entry. `mode = "none"` keeps
     the existing default-path behavior.
   - Add `mode = "auto"` to `hasPendingMetadata`.
   - Include the thumbnail prompt in `--preview-prompts`.
   - Image generation only runs when a real call is made; `--check` stays static
     and key-free.

### Quality checks
7. `scripts/check_metadata_quality.mjs`: validate `[thumbnail_generation]`
   (provider in {openai, gemini}, model non-empty string, size `WxH`, quality in
   the allowed set). Keep the existing resolved-thumbnail path checks.

### Scaffolding & secrets
8. `scripts/new_post.mjs`: change the template `[thumbnail] mode` from `none` to
   `auto` so new posts get a generated thumbnail.
9. `.op.env.example`: add `OPENAI_API_KEY=op://VAULT/ITEM/field`.

### Workflow
10. `.github/workflows/generate-metadata.yml`, modeled on
    `translate-content.yml`:
    - Trigger: `push` to `main` on `src/content/posts/**`,
      `src/content/metadata.toml`, `src/content/prompts/**`; plus
      `workflow_dispatch`.
    - Guard: skip when the head commit message contains `[skip metadata]`.
    - Concurrency group; `permissions: contents: write`.
    - Env: `GEMINI_API_KEY` and `OPENAI_API_KEY` passed directly from secrets
      (CI uses secrets directly, not `op`).
    - Run `npm run metadata:generate`, then `python3 scripts/verify.py --mode ci`.
    - Restrict committed changes to `src/content/posts/**` and
      `public/thumbnails/**`; commit with
      `chore: generate article metadata [skip metadata][skip translate]` and
      `git pull --rebase` before push to avoid racing the translate workflow.

### Docs & decisions
11. Update `.project/metadata.md` (thumbnail auto mode, OpenAI provider, keys),
    `.project/build.md` (OPENAI_API_KEY), and `.project/structure.md`
    (public/thumbnails, new workflow) as needed.
12. Add `.decisions/` entries: thumbnail image-generation model choice
    (gpt-image-2) and the OpenAI image provider.

## Verification

- `node scripts/check_frontmatter.mjs`
- `node scripts/generate_metadata.mjs --check` (static, key-free)
- `npm run metadata:preview-prompts`
- A real generation dry-run locally once `OPENAI_API_KEY` is in `.op.env`
  (`npm run metadata:generate:op`) to confirm an image is produced and wired in.
- `python3 scripts/verify.py --mode ci`
- `actionlint` on the new workflow (part of verify.py pre-push).
