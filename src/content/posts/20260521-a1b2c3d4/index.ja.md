手書きの引数パースは、最初の数行は気持ちがいい。問題はその後だ。フラグが増えるたびに分岐が増え、ヘルプ文と実装がずれていく。ここでは型を一次情報として扱い、それ以外を導出する設計を辿る。

## 何が問題だったか

従来の CLI では、同じ情報を三回書いていた。引数の定義、ヘルプの説明、そして検証ロジックだ。三つはすぐにずれる。

- ヘルプに無いフラグが動く
- 動くはずのフラグがヘルプにだけ存在する
- 検証が一部のパスでしか効かない

:::warn
「とりあえず if を足す」は短期的には速い。しかしこの分岐は誰もテストしなくなり、半年後に必ず壊れる。
:::

## 型で形を与える

サブコマンドを代数的データ型として宣言する。すると網羅性チェックが効き、新しいサブコマンドを足したときにコンパイラが処理漏れを教えてくれる。

```typescript
type Command =
  | { kind: "build"; target: string; release: boolean }
  | { kind: "watch"; paths: string[] }
  | { kind: "clean" };

function run(cmd: Command) {
  switch (cmd.kind) {
    case "build": return build(cmd.target, cmd.release);
    case "watch": return watch(cmd.paths);
    case "clean": return clean();
  }
}
```

## ヘルプ・補完・検証を導出する

型が一つあれば、残りは関数として導ける。順番はこうだ。

1. 型からスキーマを生成する
2. スキーマからヘルプ文を組み立てる
3. スキーマからシェル補完を出力する
4. 入力をスキーマで検証してから型へ写す

:::info
この方式の利点は、ヘルプと実装が構造的にずれられないことだ。両方が同じ型から生えている。
:::

## 結果

手書きの分岐は消え、コードは半分以下になった。何より、フラグを足すのが怖くなくなった。コンパイラが付き添ってくれるからだ。
